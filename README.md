# AidLockIn

**Allocate scarce disaster aid once. Prove it everywhere.**

AidLockIn gives every partner organization in a disaster response (NGOs, government agencies, donors) a shared, real-time view of what a household has already received — so the same family can't draw two food packs from two different NGOs, and the last shelter bed can't go to two people at once. Built for a hackathon on the AWS Databases + Vercel integration track: Next.js on Vercel, Amazon Aurora DSQL as the database.

The hero feature isn't "fast database." It's that two independent organizations, racing to allocate the exact same scarce resource at the exact same instant, get a *correct* outcome — enforced by the database itself, not by application-level locking that a second server or a network blip could get around.

## How the core guarantee works

Two mechanisms, both enforced by Postgres-level constraints rather than app code:

- **No duplicate entitlements.** Every aid type has a policy ("one food pack per household per 24 hours," "one shelter bed at a time"). Each allocation attempt tries to `INSERT` into `entitlement_claims` under a unique index on `(event_id, household_id, aid_type_id, policy_window_bucket)`. If a household already has a claim in that window, the insert collides with the unique index and the database itself rejects the duplicate — there's no race condition to get right in JavaScript.
- **No double-spend of a scarce resource.** Fungible stock (food, vouchers) is decremented with `UPDATE resource_pools SET remaining_quantity = remaining_quantity - 1 WHERE id = $1 AND remaining_quantity > 0`. Named units (a specific shelter bed) are claimed with a conditional `UPDATE ... WHERE status = 'AVAILABLE'`. Either way, if two requests hit the same row at once, exactly one `UPDATE` affects a row and the other affects zero — that's the whole mechanism. On Aurora DSQL, the loser actually gets retried automatically and then cleanly denied once it sees the winner's committed state; see `lib/db.ts` and `lib/allocation.ts` for the details.

Every approval and denial is also written to an append-only, hash-chained audit log (`lib/audit.ts`) the moment it happens, so a denial is just as permanently recorded as an approval — and the chain is independently re-verifiable from `/audit` or via `GET /api/audit`.

```
  db.ts                   Dual backend: Aurora DSQL or plain Postgres
  allocation.ts           The transactional core: dedup + resource claim
  overrides.ts            Coordinator override request/decision flow
  race-demo.ts            Orchestrates the two-station concurrency demo

components/               UI: shell (sidebar/topbar/role switcher), primitives
```

## Local development

### Fastest path: plain Postgres

You don't need an AWS account to develop or demo this — Aurora DSQL is wire-compatible enough with Postgres that the entire app runs unmodified against a local database.

1. `npm install`
2. Copy `.env.example` to `.env.local` and set `DATABASE_URL` to any Postgres 14+ database (a local install, Docker, Supabase, Neon, etc).
3. `npm run db:migrate` — applies `db/schema.sql`.
4. `npm run db:seed` — creates a demo disaster event ("Typhoon Nari · Kanto Response") with 6 partner organizations, 5 aid types, and a real slice of allocation history generated by actually calling the allocation engine (not fabricated rows) — including a few genuine duplicate denials and one pending + one resolved override request, so the app looks lived-in the moment you open it.
5. `npm run dev` and open `http://localhost:3000`.

### Against a real Aurora DSQL cluster

1. In the AWS Console, go to **Aurora DSQL** and create a cluster (single-Region is enough for a demo). Note its endpoint — it looks like `abc123xyz.dsql.us-east-1.on.aws`.
2. Attach an IAM policy granting your user/role `dsql:DbConnectAdmin` (and `dsql:DbConnect` if you create a non-admin database user) on that cluster's ARN.
3. Set in `.env.local`:
   ```
   DSQL_ENDPOINT=abc123xyz.dsql.us-east-1.on.aws
   PGUSER=admin
   AWS_REGION=us-east-1
   ```
4. Make sure your local AWS credentials are available (`aws configure`, or `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY` in the environment) — the connector signs IAM auth tokens per-connection, there's no separate password.
5. `npm run db:migrate && npm run db:seed`, then `npm run dev`.

DSQL has a few hard constraints `db/schema.sql` and `scripts/migrate.ts` are written around: no foreign keys/triggers, one DDL statement per transaction, and every secondary index (unique or not) is created with `CREATE INDEX ASYNC` and polled via `sys.wait_for_job` rather than built synchronously. The migration script handles this automatically on both backends.

## Deploying to Vercel

1. Push this project to a Git repo and import it into Vercel.
2. On the project's **Integrations** tab, add the **Amazon Aurora DSQL** integration from the Vercel Marketplace. This provisions a cluster and automatically injects `PGHOST`, `PGUSER`, `PGDATABASE`, `PGPORT`, `AWS_REGION`, and `AWS_ROLE_ARN` into your project's environment — including OIDC-federated credentials, so no static AWS access keys ever sit in your Vercel environment variables.
3. After the first deploy, run the migration and seed scripts once against that cluster (`vercel env pull` to get the injected variables locally, then `npm run db:migrate && npm run db:seed`).
4. Redeploy (or just visit the site — the schema and seed data are already in place).

## Demo script

1. **Overview** — the pitch and live stats.
2. **Field allocate** — type or "scan" a claim code, pick an aid type, hit *Check & allocate*. Run it again with the *same* code and aid type immediately after: clean duplicate denial, with a *Request override* option.
3. **Race demo** — the centerpiece. Hit *Run the race*; two stations (Tokyo, Osaka) fire the same real allocation call at the same instant against the same shelter bed. Exactly one wins. Hit *Run it again* — it's fully repeatable.
4. **Overrides** — switch the role switcher to **Coordinator**, approve the pending override request created in step 2 (or from seed data).
5. **Audit log** — filter to *Overrides* or *Denied*, point at the *Verified · unbroken chain* badge, export the CSV.

## Demo-mode role switcher

There's no real authentication — the sidebar's **View as** control swaps between Field / Coordinator / Donor / Admin, which changes which organization identity API calls are made under and which UI affordances (like approving overrides) are shown. This was a deliberate scope cut for the hackathon timebox; every write still goes through the same real validation regardless of which role is "viewing."
